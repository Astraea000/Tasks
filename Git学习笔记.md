# 元谋人的Git奇妙冒险

## 一、结绳旧法方知陋，野人初醒识Git

Git，这个词是Linux的作者Linus先生自己创造的一个词，在英式俚语中意为**。当然了，Git的意义不在于词义的本身，而在于它所代表的工具。如果要用一句话描述Git的话，那就是：

**迄今为止世界上最先进高端大气上档次的分布式版本控制系统**

下面我将阐述我对于这句话的的理解。

+ **分布式版本控制系统**

  廖雪峰老师已经讲的很好了，所以我决定按自己的理解从Minecraft这款游戏的视角理解和阐述Git。所有的Minecraft玩家在第一次进入多人联机服务器时，都会下载同一个地图，这与Git的clone完整仓库完全一致，每个人都可以获取一个单独的地图。在联机的时候，大家同时对一个地图进行修改，服务器将两个内容拼在一起，这就好像Git的合并功能。两个人同时挖一个方块，服务器会进行判断，最后只保留一个操作，在Git中，冲突需要人工解决，最后也不会随意覆盖代码。在Minecraft中，没有网络，离线也可以玩，但此时对地图的改动不会传送到服务器，而只是在自己的设备中，Git也是一样，离线状态也可以进行工作，并非一定需要联网，联网只是为了远程传输文件到主服务器中去。在游玩结束后，Minecraft会对该地图进行存档，玩家也可以选择定时存档或者手动存档，Git中也是如此，只不过Git会更加细致地记录user的每一个操作，同时也可以回溯到任意一个节点，以供user进行二次修改。总结下来的话，Git就是一个可以完整记录代码每一次修改，支持多人独立开发而互不干扰，还可以回溯历史版本，进行成果合并的一个版本控制系统。



+ **迄今为止最先进高端大气上档次**

  对比SVN等集中式版本控制系统，Git可以做到服务器炸掉也不影响你写代码，更不用说可以迅速恢复。速度方面也是碾压其他版本控制系统，别的系统还要连接到服务器，而Git绝大部分内容直接在本地完成，速度超级快。拥有绝对的安全，所有的错误都可以回溯恢复再修改，不必担心做错。好吧，与其他版本操作系统对比，Git的确是最强的，至今没有哪一个工具可以全方位碾压。所以可以称之为“迄今为止最先进高端大气上档次”。

## 三、野人不识版本术，穴居初学Git招

经过简单的安装操作，我们终于“招聘”到了Git这个打工人！那么如何“玩转”Git？如何让Git高效地为我们打工？下面将一一进行阐述。

1. 让Git认识你

   用Git的母语告诉它你是谁以及如何联系到你：

   `git config --global user.name "Your Name"`

   `git config --global user.email "email@example.com"`

   这有助于远程操作显示你的“代号”——做好事要留名。

   

2. 本地仓库的设置

   - 第一种方法最简单，打开一个你想使用的文件夹（最好是英文名），右键选择open git bash here即可
   - 第二种方法：首先输入`cd ＋想设置的仓库位置`，再输入`mkdir ＋“文件名”`，再输入`cd ＋“文件存储路径“`，再输入`git init`初始化仓库，接下来就可以正常开始工作了！

   

3. 文件的添加与提交

   首先要明白的是，Git是你的仆人，所以它不能到不允许进入的地方去偷看，它只能在规定范围内活动，所以我们生产整理好的”货物“需要送到和Git配套的驿站，也就是前面创建的文件夹，毕竟顺丰驿站的快递不能送到京东的仓库，这样Git才能接收到对应的货物。扔到驿站以后，我们只需输入:

   `git add “文件名”."文件类型“`即可

   此时这个文件就进入了集中发货区，当然，这意味着它进入了运输到仓库的最后一步，究竟要不要送入仓库，还需要我们进一步筛选。为什么这样做呢？这样做其实有利于我们高效工作，把所有可能送到仓库的快递放在一起，再一次性进行挑选和发送，有利于节约时间提高效率。

   

   那么如何发送我们需要的货物呢？同样的我们只需要输入：

   `git commit -m ＋”注释“`

   为什么要写注释呢？那是因为输送每一个货物到仓库都需要写一个原因，每一条输送都有对应的原因，这样便于以后进行查找和修改。

   

   注：驿站——集中发货区——仓库，Git只能在集中发货区和仓库活动哦

   

4. 查询当前仓库状况

   作为整块领土的统治者，我们当然有权利查看仓库的当前状态，只需输入：

   `git status`

   

   这样就会显示出如下可能出现的内容：

   红色的字——该货物目前在驿站，但是还没add到集中发货区，更不能进行筛选commit到仓库

   绿色的字——该货物目前被add到了集中发货区，但是还没进行筛选commit到仓库，正在等待主人的操作

   nothing to commit——驿站没有待add的货物，集中发货区也没有待commit的货物，懒家伙，啥也没干！

   

5. 查询**The Last One**

   想看看在驿站的还没add的货物变化？很简单！只需要输入：

   `git diff`

   这样就会显示目前驿站与上次驿站的区别

   

   想看看集中发货区还没commit的货物的变化？很简单！只需要输入：

   `git diff --staged`

   这样就会显示目前集中发货区与上次集中发货区的变化

   

   想看看具体某个货物的更新与变化？很简单！只需要输入：

   `git diff ＋”文件名“`

   这样就会显示该货物目前与上次的变化，记住，**所有的都diff只能看到上一次哦！**

   

   注意！为了防止内容的丢失，建议文件每完成一阶段，就给它commit一下。

   如果你是一位每天要发上万件快递的大老板！那么diff出来的内容会很长很多，Git会自动进入翻阅模式，此时只需轻按一下q键，即可退出翻阅模式。

   

6. 查询**All**

   你说你要当诸葛孔明，事事过问，事事操心，鞠躬尽瘁，死而后已，当然没问题！获取所有的发送记录，只需要输入：

   `git log`

   这样就会显示所有提交过的记录，每一条记录会显示出以下信息：

   *commit  balabala——版本号（这个东西要记住，后面会用到）*

   *Author  balabala——你的代号以及邮箱*

   *Date  balabala——这不需要我说是什么了吧*

   *balabala——上面-m后面的内容，也就是你的修改理由*

   

   你说当一个完整的诸葛孔明好累，你只想当半个，OK，当然可以，想当几分之几都没问题！输入：

   `git log -n`

   即可查看最近n次的提交记录，想看多少内容完全凭君心情。

   

   你说你没那么多心思，不想当诸葛孔明，只想学陶渊明，平平淡淡才是真，不愿纷繁复杂进入视线，当然没问题！输入：

   `git log --oneline`

   这样查询出来的内容就只会显示一条了，是不是很简洁？

   

   你说你心情好的时候是诸葛孔明，心情差的时候是陶渊明，你想二者兼具，当然没问题！输入：

   `git log -n --oneline`

   看到的内容包你满意哦。

   同样的，log出来的内容太多也会自动进入翻阅模式，轻按q键，即可退出翻阅模式。

   

7. 版本回退

   假设某件货物已经经过了n次commit（记住喽，只有commit的东西才是一个版本，add啥的都不算哦），可是你施加咒语，让时间逆转，将该货物变回了之前的某一版本，此时你需要怎么施加咒语呢？

   如果某文件的commit历史分别是A（最初版本）、B（加入新功能）、C（又改了新东西）三个版本，并且当前版本在C版，那么分别有以下几种版本回退的咒语：

   - `git reset --mixed "A的版本号"`

     执行之后呢，版本库回到A版本，ABC三个版本的代码都存着在，但是B和C被打回驿站，变成了还未add的文件。此乃温柔回退，一般这种最常用。

     

   - `git reset --soft "A的版本号"`

     执行之后呢，版本库依旧回到A的版本，ABC三个版本的代码都存着在，但是B和C被打回集中发货区，变成了已经add过但是还没有commit的文件。此乃更温柔回退。

     

   - `git reset --hard "A的版本号"`

     执行之后呢，版本库还是回到A的版本，但是A版往后的版本就都被删除了。此乃暴力回退。

     

   总结：不论是哪种回退，至少commit都会被清退，因为版本回退针对的是已经commit过的东西。对应回退的版本之后的内容留下还是抛弃，那就取决于你使用温柔还是暴力的方法了。

   

8. 管理修改

   码字其实是一件很累的事情，经过前面的解释，这个部分我决定用一句话解释，那就是：

   **对同一个文件而言，多次add会将文件进行累计合并，Git只会commit最后一次add过的文件。**

   

9. 撤销修改

   用人话说就是把文件恢复到**上一个**版本，只不过撤销修改不局限于commit过的文件，分三种情况，以下分别进行解释：

   - 该文件只进行了修改，还没有进入集中发货区，也就是还没来得及add，请输入：

     `git checkout -- "文件名"`

     

   - 该文件被add到集中发货区，还没来得及commit，请输入：

     `git reset HEAD "文件名"`

     此时文件会被退货到驿站，成为尚未add的文件，再参考第一种情况进行修改。

     

   - 已经commit到仓库，此情况请参考第7条的版本回退方法，再根据上述两条方法进行修改。

   

10. 删除文件

    分两种情况，一种是彻底删除：

    - 首先在电脑中删掉，接着让Git知道它已经被删掉，此时输入：

      `git add "文件名"`

      `git rm "文件名"`

      最后再输入：

      `git commit -m "注释"`

      OK，该文件已经彻底消失

    另一种情况，文件在电脑保留，但是不再让Git管理：

    - 直接输入：

      ``git rm --cached "文件名"`

    注意！只要还没有commit删除，都可以通过输入`git checkout -- "文件名"`找回。

    

好了，Git的基本使用方法都已经进行了阐述，如果理解到位的话，后续内容就不需要过多的比喻了，下面我们进入远程仓库的内容介绍。



## 四、孤篇难济群英力，远库方容众匠工

这一部分主要介绍有关远程仓库与分支的问题。

1. 添加远程仓库

   首先在GitHub上建立仓库，再输入

   `git remote add origin "网址"`

   即可关联，要注意的是，origin是远程仓库的小名，只是默认叫这个,你也可以取其他的名字。

   如果要取消关联的话，请输入：

   `git remote rm origin`即可。

   

2. 提交到远程仓库

   这个提交要注意，提交的是commit过的内容，第一次提交需要写：

   `git push -u origin main`

   以后提交只需要写：

   `git push`

   注意！第一次可能会有一个SSH警告，确认即可。但是我在进行提交的时候是需要输入一个验证码进行确认。

   

3. 远程库的克隆

   第一次找到一个库，我想全部下载下来进行开发，输入：

   `git clone "网址"`

   即可获取全部内容。

   这个库有了更新，我只需要获取新的内容，输入：

   `git pull origin "分支名"`

   相当于`git fetch`+`git merge`，也就是先下载到本地，再合并到分支，merge的相关知识在第4条。

   

4. 分支的创建与合并

   分支就好像代码的副本，用来独立安全地进行开发，下面介绍有关分支的相关语法：

   查看当前分支：`git branch`

   新建分支：`git branch "分支名字"`

   切换分支：`git switch "分支名字"`

   创建加切换：`git switch -c switch "分支名字"`

   合并到当前分支：`git merge "分支名字"`

   删除本地分支：`git branch -d "分支名字"`

   给分支改名：`git branch -m "分支名字"`

   诶哟记起来好麻烦，以后有需要的话再来找吧......

   

5. 解决冲突

   在我们进行pull或者merge时，有时候代码会因为互相覆盖等原因产生冲突，此时就需要我们手动进行调解。

   首先在Git里面输入pull或者merge语法时产生冲突，此时我们打开文件，会发现代码被>>>、<<<、===三种冲突字符隔断开来，需要记住，<<<和>>>之间的内容是冲突部分，而===把网上和本地的代码分割开来，本地内容靠近<<<，而网上内容靠近>>>，比如：

   <<<本地===网上>>>

   看起来很像书名号吧，如果你需要留下本地内容，那么请删除网上部分，再删掉所有多余的<<<===>>>符号，留下网上内容则与之对应，如果都需要的话，直接删掉多余的冲突符号。

   确保没有冲突符号后，再按照add，commit，push的步骤推送到网上去。

   记住了！一定一定要把冲突符号删干净，不然Git会认为你还有冲突没解决哦！

   OK，一个冲突就这样人工解决了！

   

6. 如何使用Git在GitHub协同合作

   不废话，直接上流程：

   项目所有者把你加入——clone 仓库 到本地——每次更新pull最新代码—— 新建分支写代码——add，commit，push到自己的GitHub——若有冲突则手动修改，若无则这一步省略——提交PR给作者合并

   

7. 提交hello作业的流程

   不废话，直接上流程：

   cd到该文件夹——git init——add、commit——关联网页——push  -u——成功上传

   要注意的一点是，多个源进行更改时，需要拉取最新仓库之后再上传。
